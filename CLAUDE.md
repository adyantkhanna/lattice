# CLAUDE.md — Rules of the Road for Lattice

This file is read by Claude Code at the start of every session. Follow
every rule here without being asked.

---

## 1. Milestone discipline

**Never implement code from a later milestone before the current one is
complete.** The build order in PROJECT.md is strict. If a task would
require touching milestone N+1 code, stop and flag it.

At the end of every milestone, STOP and wait for explicit user
confirmation before starting the next milestone. Do not begin the next
milestone proactively, even if the current one finishes cleanly. Also,
after each milestone, run the full local pipeline (`pnpm install`,
`pnpm dev` to verify the app boots, `pnpm test`, `pnpm lint`,
`pnpm typecheck`) and report results before stopping.

Current milestone: **0 — Foundation** (not yet started).

---

## 2. File placement

| What | Where |
|---|---|
| DB queries | `/lib/db/queries/` only — no inline SQL/ORM calls anywhere else |
| Agent prompts | `/lib/agent/prompts/*.md` — never inline strings in TS |
| Auth logic | `/lib/auth/` — local-mode bypass lives here, not in middleware.ts directly |
| Source adapters | `/lib/sources/<adapter>.ts` |
| Pack loader | `/lib/pack-loader/` |
| UI components | `/components/<domain>/` |
| Next.js routes | `/app/` (App Router only — no pages/) |
| Tests | Next to source as `*.test.ts` |

---

## 3. TypeScript

- `strict: true` in tsconfig — no exceptions
- No `any` without an explanatory comment
- Prefer `type` over `interface` for plain data shapes; `interface` for
  things that will be extended
- Named exports everywhere; no default exports except Next.js page/layout
  files (required by the framework)

---

## 4. Environment variables

Every env var must appear in `.env.example` with a comment explaining
what it does and whether it is required or optional. Never hardcode
secrets or URLs.

Key vars:
```
AUTH_MODE=local|multi          # Controls auth bypass vs full Auth.js
ANTHROPIC_API_KEY=             # BYOK in local mode
EXA_API_KEY=                   # Web search fallback
DATABASE_URL=                  # Postgres URL (multi mode only)
NEXTAUTH_SECRET=               # Auth.js secret (multi mode only)
```

---

## 5. Database

- ORM: Drizzle. Schema lives in `/lib/db/schema.ts`.
- Local mode: SQLite at `./data/lattice.db` (gitignored)
- Multi mode: Postgres (same schema, different Drizzle client config)
- All queries go through `/lib/db/queries/*.ts` — one file per entity
- Migrations generated by `drizzle-kit generate`, never hand-edited
- `user_id = 'local'` is threaded through every query in local mode;
  never hardcode it outside of the auth stub

---

## 6. Auth

- Local mode: middleware injects `{ user: { id: 'local' } }` stub session
- Multi mode: Auth.js v5, email magic link + GitHub OAuth
- The conditional lives in `/lib/auth/middleware.ts` keyed on `AUTH_MODE`
- No auth-specific UI is shown in local mode

---

## 7. Agent architecture

```
orchestrator.ts
  └─ decompose.ts    (Haiku 4.5  — question → sub-queries)
  └─ [source adapters fetch results]
  └─ synthesize.ts   (Sonnet 4.6 — sub-results → answer + citations)
  └─ extract-nodes.ts (Sonnet 4.6 — structured concept extraction)
```

- Model IDs: synthesis = `claude-sonnet-4-6`, decomposition = `claude-haiku-4-5-20251001`
- All Anthropic calls go through the Anthropic SDK with prompt caching
  enabled where possible
- Streaming via Vercel AI SDK (`streamText` / `streamObject`)
- Source fetch results are cached in `source_cache` table; TTL TBD per adapter

---

## 8. Topic Packs

- YAML files in `/packs/<slug>.yaml`
- Validated against `/packs/pack.schema.json` at load time and in CI
- Pack loader: `/lib/pack-loader/load.ts`, `/lib/pack-loader/validate.ts`
- Never access pack files directly outside the pack loader
- Community packs are added via PR; validate-packs CI action must pass

---

## 9. Linting & formatting

- **Biome** for everything — no ESLint, no Prettier
- `biome.json` at repo root
- Runs on pre-commit via **lefthook**
- CI also runs `pnpm lint` and `pnpm typecheck`

---

## 10. Testing

- **Vitest** — `*.test.ts` files next to source
- Unit tests for: pack loader, source adapters, decompose, synthesize,
  extract-nodes, DB query helpers
- Integration tests for the agent loop (use fixtures, not live APIs)
- Run with `pnpm test`

---

## 11. Git hygiene

- **Conventional Commits**: `feat:`, `fix:`, `docs:`, `chore:`, `test:`,
  `refactor:`
- Branch naming: `feat/<name>`, `fix/<name>`, `docs/<name>`, `chore/<name>`
- Never commit directly to `main`
- PR required; CI must be green before merge

---

## 12. UI conventions

- Component library: **shadcn/ui** (add components with `pnpm dlx shadcn@latest add`)
- Styling: **Tailwind CSS** only — no inline styles, no CSS modules
- Fonts: Source Serif Pro (long-form content), Inter (UI chrome)
- Dark mode is the default; use Tailwind `dark:` classes
- Three-pane layout on desktop: knowledge tree | chat | source preview
- Citations must be rendered as hyperlinks — never bare text

---

## 13. What NOT to do

- Do not skip milestones or implement speculative future features
- Do not write inline SQL — use the query helpers
- Do not hardcode prompts in TS files — put them in `.md` files
- Do not add `any` without justification
- Do not add comments that describe *what* code does — only *why* when
  non-obvious
- Do not create planning or analysis `.md` files beyond what is specified
  in the brief (PROJECT.md, CLAUDE.md, docs/, README, etc.)
- Do not push to `main` directly
- Do not amend published commits

---

## 14. Package manager

**pnpm** only. Never run `npm install` or `yarn add`.

---

## 15. Asking before acting

For any action that is hard to reverse (deleting files, force-pushing,
dropping DB tables, modifying CI pipelines), ask the user first.

---

## 16. Topic Packs during development

Topic Pack YAMLs may exist as minimal placeholders (slug, name,
description, empty source arrays) during development. The pack loader,
validator, and agent must all handle empty/minimal packs gracefully. The
agent falls back to general web search via Exa when a pack has no
curated sources. Never generate fake pack content to fill in gaps —
leave fields empty if the user hasn't provided them.

---

## 17. Honesty about progress

When reporting milestone completion, be precise. If something was
stubbed, skipped, or doesn't work end-to-end, say so explicitly. Do not
describe a stubbed implementation as "complete" or "working". The user
is reviewing each milestone and false completion claims waste a
debugging cycle.
